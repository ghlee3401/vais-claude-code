Part 1 — Claude Code 기본 개념

Claude Code는 Anthropic의 공식 CLI 도구로, 터미널에서 Claude AI와 대화하며 소프트웨어 개발 작업을 수행합니다. 파일 읽기/쓰기, 코드 검색, 명령 실행 등의 도구(Tool)를 사용하여 실제 코드베이스와 상호작용합니다.

CLAUDE.md — 프로젝트 지침 파일

프로젝트 루트의 CLAUDE.md는 Claude가 매 대화 시작 시 자동으로 읽는 설정 파일입니다. 프로젝트의 규칙, 구조, 컨벤션을 여기에 정의하면 Claude가 이를 따릅니다.

# My Project

## What This Project Is
Next.js 14 기반 SaaS 대시보드.

## Project Structure
src/
├── app/          # Next.js App Router
├── components/   # React 컴포넌트
├── lib/          # 유틸리티, API 클라이언트
└── types/        # TypeScript 타입 정의

## Mandatory Rules
1. TypeScript strict mode 사용
2. Tailwind CSS만 사용 (인라인 스타일 금지)
3. 커밋 전 `npm run lint` 필수

## Do NOT
- node_modules/ 수정 금지
- .env 파일에 시크릿 하드코딩 금지
핵심 포인트 CLAUDE.md에 적은 규칙은 Claude의 모든 응답에 적용됩니다. "기획 없이 코드 금지", "테스트 필수" 같은 규칙을 여기에 정의하면 Claude가 자율적으로 이를 지킵니다.

Memory — 대화 간 기억 시스템

Memory는 대화가 끝나도 유지되는 파일 기반 기억 시스템입니다. 사용자의 선호, 프로젝트 컨텍스트, 피드백을 기록하여 다음 대화에서 활용합니다.

메모리 구조

memory/
├── MEMORY.md              # 인덱스 (Claude가 매번 자동 로드)
├── user_background.md     # 사용자 정보
├── feedback_testing.md    # 피드백/규칙
└── project_current.md     # 프로젝트 상태

메모리 타입 4가지

타입용도예시
user사용자의 역할, 선호, 전문성"백엔드 10년차, React는 처음"
feedback교정/확인된 작업 방식"DB mock 금지 — 프로덕션 장애 이력"
project프로젝트 상태, 결정사항"4/5까지 머지 프리즈"
reference외부 리소스 위치"버그 트래킹: Linear INGEST 프로젝트"

메모리 파일 작성 형식

---
name: 테스트 DB mock 금지
description: 테스트에서 DB를 mock하지 말고 실제 DB를 사용해야 한다는 규칙
type: feedback
---

테스트에서 DB를 mock하지 않는다. 항상 실제 DB(테스트 인스턴스)에 연결한다.

**Why:** mock과 실제 DB 동작 차이로 인해 테스트 통과 → 프로덕션 장애 발생 이력 있음.

**How to apply:** 테스트 파일 작성 시 DB 관련 의존성은 mock하지 않는다.
MEMORY.md 인덱스 MEMORY.md는 각 메모리 파일의 한 줄 요약 목록입니다. Claude는 매 대화 시작 시 이 인덱스를 읽어 전체 기억을 빠르게 파악합니다. 200줄 이후는 잘리므로 간결하게 유지하세요.

Hooks — 이벤트 기반 자동화

Hooks는 특정 이벤트 발생 시 자동으로 실행되는 셸 스크립트입니다. Claude의 행동을 프로그래밍적으로 통제하는 핵심 메커니즘입니다.

Hook 이벤트 종류

이벤트시점활용 예
SessionStart대화 시작 시상태 복원, 컨텍스트 주입
PreToolUse도구 실행 직전위험 명령 차단 (bash-guard)
PostToolUse도구 실행 직후파일 변경 추적 (doc-tracker)
Stop응답 완료 시체크포인트 저장
SubagentStart하위 에이전트 시작에이전트 상태 기록
SubagentStop하위 에이전트 종료산출물 검증, Gate 체크

hooks.json 설정 예시

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node scripts/bash-guard.js",
            "timeout": 3000
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "node scripts/doc-tracker.js",
            "timeout": 5000
          }
        ]
      }
    ]
  }
}
Hook의 exit code가 통제를 결정합니다 exit(0) = 허용, exit(2) = 차단. PreToolUse 훅에서 exit(2)를 반환하면 해당 도구 호출이 거부됩니다. 이것이 AI를 프로그래밍적으로 통제하는 핵심 원리입니다.

Agents — 역할별 하위 에이전트

Agent는 특정 역할을 수행하는 독립 서브프로세스입니다. 마크다운 파일에 frontmatter로 정의하며, Agent() 도구로 호출합니다.

---
name: executor
version: 1.0.0
description: |
  실행 에이전트. 구체적 구현 작업을 담당합니다.
  Triggers: (직접 호출 금지 — orchestrator를 통해 호출)
model: sonnet
tools: [Read, Write, Edit, Glob, Grep, Bash, TodoWrite]
disallowedTools:
  - "Bash(rm -rf*)"
  - "Bash(git push*)"
  - "Agent"           # 하위 에이전트 호출 불가
---

# Executor Agent

당신은 구현 전담 에이전트입니다.

## 구현 원칙
- 최소 변경: 요청된 기능만 구현
- 테스트 포함: 구현과 동시에 단위 테스트 작성

Frontmatter 핵심 필드

필드설명
model사용 모델 (opus: 전략/판단, sonnet: 실행/구현)
tools허용 도구 목록
disallowedTools차단 도구 패턴 (안전장치)
memory메모리 접근 수준 (read-write, project, none)

Skills — 슬래시 커맨드

Skill은 /명령어로 호출하는 재사용 가능한 작업 단위입니다. SKILL.md에 정의하며, 인자를 받아 동적으로 동작합니다.

/example plan login      → 기획서 작성
/example implement login → 구현
/example review login    → 검토 + Gap 분석

SKILL.md 내부에서 $0은 액션, $1은 피처명으로 치환됩니다.

MCP — 외부 도구 연결

Model Context Protocol은 Claude가 외부 시스템(DB, API, 파일 서비스 등)과 통신하는 표준 프로토콜입니다. .mcp.json에 설정합니다.

// .mcp.json
{
  "mcpServers": {
    "my-api": {
      "type": "stdio",
      "command": "node",
      "args": ["mcp/my-server.js"],
      "env": { "API_KEY": "${API_KEY}" }
    }
  }
}

MCP 서버를 통해 Claude는 프로젝트 외부의 데이터에 접근하거나 외부 서비스를 호출할 수 있습니다.

Part 2 — Harness Engineering

Harness Engineering은 Claude Code 위에 에이전트 조직과 자동화 워크플로우를 구성하는 설계 패턴입니다. AI에게 "무엇을 하라"가 아니라 "어떤 구조 안에서 일하라"를 정의합니다.

📐
구조로 통제
프롬프트가 아닌 파일 구조와 훅으로 AI의 행동 범위를 결정
🔒
안전장치 내장
위험 명령 차단, 워크플로우 순서 강제, 산출물 검증 자동화
🏗️
역할 분리
Orchestrator는 전략, Executor는 실행. 계층 구조로 책임 분리
핵심 원리 Claude Code의 CLAUDE.md + hooks + agents를 조합하면, AI가 정해진 워크플로우 안에서만 동작하도록 하는 "하네스(마구)"를 만들 수 있습니다. 마치 말에 마구를 씌워 방향을 잡듯, AI에게 구조를 씌워 생산성을 극대화합니다.

Orchestrator Pattern — 계층적 에이전트

하네스의 핵심은 Orchestrator-Executor 계층 구조입니다. 전략적 판단은 Orchestrator가, 구체적 실행은 Executor가 담당합니다.

Orchestrator (Opus) ——위임——> Executor (Sonnet)
OrchestratorExecutor
모델Opus (고성능, 판단력)Sonnet (빠른 실행)
역할워크플로우 관리, 에이전트 호출, 품질 게이트코드 작성, 테스트, 구현
도구Agent 도구 포함 (하위 에이전트 호출 가능)Agent 도구 없음 (실행만)
직접 호출가능 (사용자 또는 스킬에서)금지 (Orchestrator를 통해서만)
왜 분리하나? Opus 모델이 모든 코드를 직접 쓰면 비용이 높고 느립니다. 전략적 판단만 Opus가 하고, 실제 코드 작성은 Sonnet에게 위임하면 비용 효율과 속도를 모두 얻습니다. 또한 Executor에서 Agent 도구를 차단하면 하위 에이전트가 무한히 에이전트를 호출하는 것을 방지합니다.

Hook 자동화 — AI 행동의 프로그래밍적 통제

Hook은 하네스의 근육입니다. 이벤트 기반으로 AI의 행동을 자동 통제합니다.

SessionStart session-start.js — 이전 대화 상태 복원, 진행 중인 피처/단계 컨텍스트 주입
PreToolUse(Bash) bash-guard.js — rm -rf, git push --force, DROP TABLE 등 위험 명령 차단
PostToolUse(Write|Edit) doc-tracker.js — docs/ 하위 파일 변경 이력 기록
SubagentStart agent-tracker.js — 에이전트 시작 기록, 상태 파일 업데이트
SubagentStop agent-tracker.js — 에이전트 종료 기록, 산출물 검증
Stop stop-handler.js — 체크포인트 저장 (다음 대화에서 이어받기 가능)

Safety Guard — 위험 차단

bash-guard.js는 Bash 명령 실행 전에 위험 패턴을 검사합니다. 차단(BLOCK)과 경고(WARN) 두 단계로 동작합니다.

// bash-guard.js 핵심 로직

const BLOCKED = [
  { pattern: /rm\s+-rf\s+[^-]/,               reason: '재귀 삭제' },
  { pattern: /git\s+push\s+--force/,           reason: '강제 푸시' },
  { pattern: /DROP\s+TABLE/i,                  reason: 'DB 테이블 삭제' },
  { pattern: />\s*\/dev\/sd[a-z]/,             reason: '디스크 직접 쓰기' },
];

const ASK = [
  { pattern: /rm\s+-r\b/,                      reason: '재귀 삭제 — 확인 필요' },
  { pattern: /git\s+reset\s+--hard/,           reason: '변경사항 소실 위험' },
];

// BLOCKED → exit(2)로 즉시 차단
// ASK → 경고 메시지 + 사용자 확인 요청
BLOCKED vs ASK BLOCKED 패턴은 exit(2)로 즉시 거부됩니다. AI가 아무리 정당한 이유를 대도 실행 불가. ASK 패턴은 경고를 표시하고 사용자 확인을 요청합니다. 이 이중 안전장치가 프로덕션 사고를 방지합니다.

Part 3 — VAIS Code

VAIS Code(Virtual AI C-Suite)는 하네스 엔지니어링의 개념을 기업 조직 구조로 확장한 Claude Code 플러그인입니다. CEO, CTO, CSO 등 C-Level 에이전트가 각자의 전문 영역을 담당하며, 서비스 런칭의 전체 라이프사이클을 자동 실행합니다.

🏢
basic/ → VAIS
Orchestrator 1개 → C-Suite 7개. Executor 1개 → 실행 에이전트 14개. 역할 세분화
🛡️
7개 품질 게이트
CTO Gate 1-4 + CSO Gate A/B/C. 프로그래밍적 자동 검증
⏸️
체크포인트 멈춤
중요 결정마다 반드시 사용자에게 확인. 자동 진행 금지

체크포인트 멈춤 — 사용자와의 소통

VAIS Code의 가장 중요한 통제 메커니즘은 체크포인트(CP)입니다. 모든 C-Level 에이전트는 중요 결정 지점에서 반드시 멈추고 사용자에게 확인을 요청합니다.

CTO 에이전트의 체크포인트

CP시점사용자에게 묻는 것
CP-1Plan 완료 후"구현 범위를 선택해주세요" → 최소 / 표준 / 확장
CP-DDesign 완료 후"아키텍처 옵션을 선택해주세요" → 최소변경 / 클린아키텍처 / 실용적균형
CP-2Do 시작 전"다음 에이전트를 실행합니다. 실행할까요?" → 실행 / 수정 / 중단
CP-GGate 완료 후"Gate 결과입니다. 계속할까요?" → 계속 / 수정 후 재검 / 중단
CP-QQA 완료 후"QA 결과입니다." → 모두 수정 / Critical만 / 그대로 진행

체크포인트 강제력: 4중 보강

CP를 건너뛰는 것을 방지하기 위해 4가지 메커니즘이 동시에 작동합니다.

1. 에이전트 .md    → CP 테이블 + "위반 시나리오" 금지 목록
2. cp-tracker.js  → AskUserQuestion 호출 시 event-log에 기록
3. cp-guard.js    → SubagentStop에서 CP 기록 검증 → 미충족 시 exit(1)
4. auto-judge.js  → Stop 시 전체 워크플로우 흐름 평가
AI가 "자율적으로 다 해버리는" 것을 방지 체크포인트가 없으면 AI가 plan부터 deploy까지 한 번에 달려갈 수 있습니다. 사용자가 원하지 않는 방향으로 코드가 작성될 위험이 있죠. CP는 "여기서 멈추고 사용자에게 방향을 확인하라"는 강제 장치입니다.

품질 게이트 — 자동 검증

gate-check.js가 프로그래밍적으로 산출물을 검증합니다. 체크리스트 항목을 자동 확인하여 PASS/FAIL을 판정합니다.

Gate검증 항목소관
cto-1Plan 문서 존재, 데이터 모델 정의, 기술 스택 선정CTO
cto-2Design 문서 존재, Interface Contract, API 엔드포인트CTO
cto-3Architect 문서, 환경 변수 템플릿CTO
cto-4Do 문서, 피처 레지스트리 구현율 ≥ 90%CTO
cso-a보안 문서, Critical 0건, OWASP ≥ 8/10, 배포 승인CSO
cso-bpackage.json, claude-plugin, 버전 동기화, SKILL.mdCSO
cso-cCSO QA 문서, 품질 점수 ≥ 80/100, Critical 0건CSO

Observability — 투명한 추적

VAIS Code는 에이전트의 모든 활동을 추적합니다.

파일형식내용
.vais/agent-state.jsonJSON현재 활성 에이전트, 단계, 피처
.vais/event-log.jsonlJSONL (append-only)모든 이벤트: 에이전트 시작/종료, 게이트 체크, CP, 문서 변경
docs/{단계}/{역할}_{피처}.mdMarkdown각 단계의 산출물 문서
왜 JSONL인가? append-only 형식이므로 동시 접근에 안전하고, 로그 파일이 커져도 최신 이벤트를 빠르게 추가할 수 있습니다. 30일 이후 자동 아카이브됩니다.

FAQ

Q. 모든 C-Level을 거쳐야 하나요?

아닙니다. 기술 구현만 필요하면 /vais cto login으로 CTO만 호출하면 됩니다. 전체 런칭이 필요할 때만 CEO를 사용합니다. 개별 C-Level을 직접 호출할 수 있습니다.

Q. 체크포인트에서 매번 멈추는게 귀찮으면?

체크포인트는 "AI가 잘못된 방향으로 달려가는 것"을 방지하는 안전장치입니다. 각 CP에서 간결한 요약과 명확한 선택지를 제공하므로, 한 단어 응답으로 빠르게 진행할 수 있습니다.

Q. 기존 프로젝트에 VAIS를 적용하려면?

/vais init my-feature 를 실행하면 기존 코드베이스를 분석하여 VAIS 문서(plan, design 등)를 역생성합니다.

Q. 에이전트가 위험한 명령을 실행하려 하면?

bash-guard.js가 자동으로 차단합니다. rm -rf, git push --force, DROP TABLE 등은 AI가 어떤 이유를 대도 실행되지 않습니다. disallowedTools 설정으로 에이전트별 도구 제한도 가능합니다.

Q. basic/ 폴더로 나만의 하네스를 만들려면?

basic/ 디렉토리를 복사하여 새 프로젝트의 루트에 놓으세요. CLAUDE.md에 프로젝트 규칙을, agents/에 역할을, hooks/에 자동화를 정의하면 됩니다. Orchestrator→Executor 구조를 유지하되, 프로젝트에 맞게 에이전트를 추가/수정하세요.

Q. Opus와 Sonnet 사용을 어떻게 나누나요?

에이전트 frontmatter의 model 필드로 결정합니다. 전략적 판단(기획, 설계, 오케스트레이션)은 Opus, 실행(코드 작성, 검사, 감사)은 Sonnet을 사용합니다. 이를 통해 비용 대비 품질을 최적화합니다.
