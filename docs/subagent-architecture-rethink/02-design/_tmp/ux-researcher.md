# UX 리서치 설계: subagent-architecture-rethink

## 0. 리서치 목적 및 범위

### 목적

이 리서치는 SC-10(b)의 직접 이행이다. ideation에서 사용자 1인의 자기 통증 진술(KP-1~KP-14)이 충분한 설계 근거가 될 수 없다는 전제 아래, 외부 5~7명 인터뷰를 통해 세 가지를 검증한다.

1. Project Profile schema 12개 변수의 실제 프로젝트 분류 충분성
2. 정책 4분류(A/B/C/D)가 사용자 직관과 맞는지
3. "쓸데없이 만든다" default-execute 통증이 VAIS 사용자 전반에 보편적인지
4. 산출물 50+ 카탈로그에서 실제 사용자가 우선시하는 산출물이 무엇인지

### 범위 한정

이 문서는 설계(design) 단계 산출물이다. 실제 인터뷰 수행·분석·Profile 변수 반영은 do 단계에서 진행되며, 최종 결과는 `docs/subagent-architecture-rethink/02-design/user-interviews.md`에 기록된다.

---

## 1. 인터뷰 가설 (7개)

각 가설은 검증 방법·반증 조건·학습 가치를 명시한다.

---

### H-1. Project Profile 12변수 충분성 가설

**가설**
Project Profile schema의 12개 변수(type, stage, target_scale, geography, revenue_model, monetization_active, handles_pii, handles_payments, handles_health, cross_border_transfer, deployment.target, sla_required + seo_dependent, public_marketing)는 어떤 sub-agent 산출물을 작성해야 하는지 자동 판단하기에 충분하다.

**검증 방법**
- 인터뷰어가 참여자의 실제 진행 중 프로젝트를 12개 변수로 즉석 분류
- 분류 결과가 "이 프로젝트에 DPIA가 필요한가", "SEO Audit이 필요한가" 등 정책 B 산출물 판단과 일치하는지 참여자 본인에게 검증
- 5/7 이상 참여자에서 일치하면 충분성 확보로 간주

**반증 조건**
- 3명 이상이 "이 변수로는 내 프로젝트의 특수성을 잡을 수 없다"고 응답
- 실제 분류 후 B 정책 오판 사례가 2개 이상 발생
- "이 변수가 없으면 안 된다"는 누락 변수 언급이 2명 이상에서 반복

**학습 가치**
반증 시: schema에 변수 추가 또는 기존 변수 세분화 결정 근거. 충분 시: 12변수 확정 및 lib/project-profile.js 구현 착수 신호.

---

### H-2. 정책 4분류 직관 일치 가설

**가설**
Always(A) / Scope 기반 자동(B) / User-select(C) / Triggered(D) 4분류는 사용자가 별도 설명 없이 산출물을 보고 어느 분류에 속하는지 직관적으로 분류할 수 있다.

**검증 방법**
- 10개 산출물 카드(PEST, PRD, DPIA, Postmortem, AARRR, Vision Statement, Lean Canvas, ADR, SEO Audit, Five Forces)를 준비
- 4분류 레이블과 기준 한 줄 설명 제공 후 카드 정렬 요청
- 정답(ideation 합의 분류)과의 일치율 측정. 7/10 이상 = 직관 일치

**반증 조건**
- 5/10 이하 일치
- 참여자가 "C와 B의 차이를 모르겠다"고 3명 이상 응답
- 특정 산출물에서 분류 의견이 완전히 갈리는 경우(예: PEST를 A로 분류한 2명, C로 분류한 3명)

**학습 가치**
반증 시: 4분류 레이블명 재명명 또는 분류 축소(예: B+C 통합) 고려. 직관 불일치 산출물은 분류 재검토 대상으로 표시.

---

### H-3. Default-execute 통증 보편성 가설

**가설**
"sub-agent가 필요 여부를 묻지 않고 즉시 산출물을 만든다"는 통증(ideation KP-6, KP-7)은 VAIS를 사용하는 대부분의 사용자가 경험한다. 이 통증은 특정 개인의 선호 문제가 아니라 시스템 설계 결함이다.

**검증 방법**
- "내가 원하지 않았는데 만들어진 산출물 경험이 있는가?" 개방형 질문
- 경험 있다면: 어떤 sub-agent, 어떤 산출물, 어떤 프로젝트 맥락인지 수집
- 경험 없다면: "만약 DPIA를 SEO 전용 내부 툴 프로젝트에 만들어낸다면?" 시나리오 제시 후 반응 관찰

**반증 조건**
- 3명 이상이 "나는 이런 불편 없었다, 만들어주면 좋다"고 응답
- "어차피 내가 지워도 된다"는 무관심 응답이 지배적(4명 이상)

**학습 가치**
반증 시: default-execute 해소 우선순위를 낮추고, 산출물 품질 향상에 리소스 집중 재검토. 보편성 확인 시: SC-03 우선순위 유지 근거.

---

### H-4. 산출물 우선순위 카탈로그 가설

**가설**
50개 산출물 카탈로그 중 사용자가 실제로 가장 자주 필요로 하는 5개는 PRD, User Story, Architecture Decision Record, Lean Canvas, Persona 순이다.

**검증 방법**
- 50개 산출물 리스트를 카드 또는 화면으로 제시
- "지금 당장 자신의 프로젝트에 필요한 5개를 고른다면?" 질문
- 5명 이상 선택 = "핵심 집합", 1명 이하 선택 = "낮은 수요" 레이블

**반증 조건**
- 가정한 5개 중 3개 이상이 상위 선택에 포함되지 않음
- "이 리스트에 없는데 꼭 필요하다"는 누락 산출물이 3명 이상에서 반복 언급

**학습 가치**
반증 시: SC-04/SC-08 작업 우선순위 재정렬. 수요 낮은 산출물은 C(User-select)로 강등 또는 Phase 2로 이연 결정 근거.

---

### H-5. Sub-agent 역할 명확성 가설

**가설**
현재 VAIS 사용자는 각 sub-agent가 무엇을 만드는지 호출 전에 정확히 알지 못한다. "어떤 sub-agent를 써야 하는지 모르겠다"는 인지 부담이 "쓸데없이 많이 만들어진다"는 통증과 함께 존재한다.

**검증 방법**
- "지난 한 달 동안 VAIS에서 어떤 sub-agent를 가장 많이 호출했는가?" 질문
- "그 sub-agent가 어떤 산출물을 만드는지 사전에 알고 있었는가?" 확인
- "처음 써보는 sub-agent를 실험 목적으로 호출한 적 있는가?" 확인

**반증 조건**
- 4명 이상이 "각 sub-agent 역할이 이미 명확하다"고 응답
- "카탈로그나 문서를 보고 선택했다"는 응답이 지배적

**학습 가치**
확인 시: sub-agent별 "나는 {산출물명}을 만드는 에이전트입니다" 첫 문장 표준화 및 카탈로그 UI 우선순위 상향. 반증 시: 명명 규칙 문제가 아닌 다른 인지 부담 요인 탐색으로 전환.

---

### H-6. 단계 간 alignment 인식 가설

**가설**
VAIS 사용자는 Core(전략) → Why(타깃) → What(솔루션) → How(구현) 단계 사이 정렬 문제를 자주 경험한다. 예: "전략은 B2B 엔터프라이즈인데 UI는 개인 개발자 도구처럼 만들어졌다".

**검증 방법**
- "VAIS가 만든 산출물들 간에 서로 모순되거나 방향이 달랐던 경험이 있는가?" 개방형 질문
- 경험 있다면: 어느 단계 사이에서, 어떤 산출물들이, 어떤 방향으로 어긋났는지 수집

**반증 조건**
- 4명 이상이 "그런 모순을 경험한 적 없다"고 응답
- "어차피 내가 전체를 조율한다"는 능동 개입 패턴이 지배적

**학습 가치**
확인 시: alignment 메커니즘(α + β) 설계 우선순위 유지. 반증 시: alignment 자동화보다 사용자 가이드 강화로 접근 전환.

---

### H-7. 이상적 위임 경계 가설

**가설**
사용자는 AI sub-agent에게 "완전히 위임"하고 싶은 영역과 "반드시 자신이 결정"하고 싶은 영역이 명확히 다르다. 이 경계는 개인 스타일이 아니라 산출물의 특성(정량적/정성적, 외부 공개/내부 참조, 되돌릴 수 있음/없음)에 의해 결정된다.

**검증 방법**
- "VAIS의 어떤 산출물은 완전히 믿고 그대로 쓰는가? 어떤 산출물은 반드시 직접 검토하는가?" 질문
- 두 집합의 공통 특성 귀납적 도출
- 5명의 응답에서 공통 패턴 3개 이상 추출 시 가설 확인

**반증 조건**
- 응답이 완전히 개인화되어 공통 패턴 추출 불가
- "모든 산출물을 직접 검토한다" 또는 "모든 산출물을 그대로 쓴다"는 극단적 응답이 지배적

**학습 가치**
확인 시: 산출물 메타데이터에 "user_review_required: bool" 필드 추가 근거. 위임 가능 산출물의 auto-judge 기준 강화 방향 수립.

## 2. 인터뷰 스크립트 (45분 세션)

### 사전 준비물

- 참여자 동의서 (녹음 + 익명 처리 조건)
- 산출물 카드 10개 (카드 정렬 세션용): PEST, PRD, DPIA, Postmortem, AARRR, Vision Statement, Lean Canvas, ADR, SEO Audit, Five Forces
- 4분류 레이블 카드: Always / Scope / User-select / Triggered (기준 한 줄 설명 포함)
- 50개 산출물 목록 (화면 또는 인쇄, 이름+한 줄 설명 포함)
- Project Profile 12변수 체크시트

---

### 섹션 A. 도입 (5분)

**목적**: 참여자가 편안하게 대화할 수 있도록 맥락 설정. 평가가 아님을 강조.

**진행 스크립트**
"오늘 인터뷰에 참여해주셔서 감사합니다. 저는 VAIS Code라는 AI 개발 지원 플러그인의 UX를 개선하는 작업을 하고 있습니다. 오늘은 도구를 테스트하는 것이 아니라 여러분의 실제 경험을 듣는 자리입니다. 정답이 없고 솔직한 이야기가 가장 도움이 됩니다. 혹시 이야기 흐름에서 갑작스럽게 방향이 바뀌는 경우가 있어도 모두 의도된 것이니 편하게 따라와 주시면 됩니다."

**Q-A1.** 본인을 간단히 소개해주세요. 어떤 프로젝트를 주로 만드시나요?
- 후속: 혼자 하시나요, 팀으로 하시나요?
- 후속: 지금 가장 집중 중인 프로젝트가 있다면 한 줄로 설명해주실 수 있나요?

**관찰 포인트**
- 프로젝트 유형(SaaS/Internal/OSS/API) 파악
- VAIS 사용 경험 여부 사전 확인
- 커뮤니케이션 스타일(기술 중심 vs 비즈니스 중심) 파악

---

### 섹션 B. 현재 사용 패턴 (10분)

**목적**: JTBD 배경 질문. 어떤 맥락에서 AI agent 도구를 쓰는가, VAIS 호출 패턴은 무엇인가.

**Q-B1.** AI 기반 개발 도구(Claude, VAIS, Cursor, Copilot 등)를 어떤 상황에서 가장 많이 켜시나요?
- 후속: 막히는 상황인가요, 아니면 빠르게 초안을 만들고 싶을 때인가요?
- 후속: 혼자서 할 수 있지만 시간을 아끼려고 쓰시나요, 아니면 모르는 영역이라 도움이 필요해서 쓰시나요?

**Q-B2.** VAIS를 주로 어떤 phase에서 쓰시나요? (기획 / 설계 / 구현 / 배포 / 운영)
- 후속: 가장 자주 쓰는 C-Level은 무엇인가요? (CPO / CTO / CBO / CSO / COO / CEO)
- 후속: 그 이유가 있다면?

**Q-B3.** 최근 한 달 안에 VAIS에서 가장 많이 호출한 sub-agent 3개를 떠올릴 수 있으신가요?
- 후속: 각 sub-agent가 어떤 산출물을 만드는지 호출 전에 알고 계셨나요?
- 후속: 처음 써보는 sub-agent를 실험 목적으로 호출해본 적 있나요? 그 결과는 어땠나요?

**관찰 포인트**
- H-5(역할 명확성) 가설 사전 신호 수집
- 사용 빈도가 높은 sub-agent 영역(기획/기술/비즈니스) 파악
- 도구를 "도움"으로 쓰는가 "위임"으로 쓰는가

---

### 섹션 C. 통증 발굴 (15분)

**목적**: "쓸데없이 만든다" default-execute 통증의 보편성 검증(H-3). 어떤 산출물이 부족하고 어떤 산출물이 과잉인가.

**Q-C1.** VAIS가 만들어준 산출물 중 "이건 우리 프로젝트에 필요 없는데..."라고 느낀 적 있나요?
- 후속: 어떤 sub-agent였나요? 어떤 산출물이었나요?
- 후속: 그때 어떻게 하셨나요? (삭제했다 / 그냥 뒀다 / 수정했다 / 다시 명령했다)
- 후속: 그 경험이 얼마나 불편했나요? 1~5점으로 표현한다면?

**Q-C2.** 반대로 "이 산출물이 있었으면 좋겠는데 VAIS가 안 만들어준다"는 경험이 있나요?
- 후속: 어떤 산출물이었나요?
- 후속: 직접 만드셨나요, 아니면 포기하셨나요?
- 후속: 그 산출물이 없어서 프로젝트에 어떤 영향이 있었나요?

**Q-C3.** VAIS가 만들어준 산출물이 "내 프로젝트 맥락을 전혀 모르고 만든 것 같다"고 느낀 적 있나요?
- 후속: 어떤 상황이었나요?
- 후속: 그런 일이 생기는 이유가 무엇이라고 생각하시나요?
- 후속: 만약 VAIS가 프로젝트 시작 전에 "이 프로젝트는 어떤 종류입니까, 어느 단계입니까, 사용자 규모는?"을 물어봐서 미리 파악한다면, 이 문제가 해결될 것 같나요?

**Q-C4.** 만약 VAIS가 sub-agent 호출 전에 "이 산출물이 현재 프로젝트에 필요한지 먼저 판단하겠습니다. 필요 없으면 건너뛰겠습니다"라고 한다면 어떨 것 같나요?
- 후속: 어떤 상황에서 그 자동 판단을 신뢰할 수 있나요?
- 후속: 어떤 상황에서는 내가 직접 결정하고 싶나요?

**관찰 포인트**
- H-3(default-execute 통증 보편성) 직접 검증
- H-7(이상적 위임 경계) 초기 신호 수집
- "필요 없는" 산출물과 "부족한" 산출물 목록 수집
- 자동 판단에 대한 신뢰/불신 임계점

---

### 섹션 D. 정책 4분류 + Profile 변수 검증 (8분)

**목적**: H-1(Profile 충분성), H-2(정책 직관 일치) 정량 검증.

**[카드 정렬 활동]**
"이제 잠깐 활동 하나를 하겠습니다. 10개 산출물 카드를 드릴게요. 그리고 4개 카테고리 레이블도 드릴게요."

레이블 설명:
- **Always**: 어떤 프로젝트든 무조건 만들어야 하는 것
- **Scope**: 프로젝트 종류에 따라 만들 수도, 안 만들 수도 있는 것
- **User-select**: 같은 목적을 가진 여러 방법 중 하나 골라 쓰는 것
- **Triggered**: 특정 사건이 발생했을 때만 만드는 것

"10개 카드를 이 4개 카테고리에 놓아주세요. 직관대로 하시면 됩니다."

**Q-D1.** [카드 정렬 완료 후] 분류하기 어렵거나 망설였던 카드가 있었나요?
- 후속: 어떤 점에서 애매했나요?

**Q-D2.** [Project Profile 체크시트 제시] 이 12개 질문으로 본인 프로젝트를 분류해 주시겠어요?
- 후속: [분류 완료 후] 이 분류를 보고 DPIA(개인정보 영향평가)가 필요한지 아닌지 바로 알 수 있을 것 같나요?
- 후속: 이 질문 중 "이런 걸 왜 물어보지?" 하고 이상하게 느껴진 항목이 있나요?
- 후속: 반드시 있어야 하는데 빠진 질문이 있다고 느끼시나요?

**관찰 포인트**
- 카드 정렬 일치율 기록 (정답 대비)
- Profile 변수 중 혼란을 주는 항목 식별
- 누락 변수 언급 수집

### 섹션 E. 카탈로그 우선순위 (5분)

**목적**: H-4(산출물 우선순위) 검증.

**Q-E1.** [50개 산출물 목록 제시] 지금 당장 진행 중인 프로젝트에 필요한 산출물 5개를 골라주세요. 제한 시간은 2분입니다.

**Q-E2.** [선택 완료 후] 가장 먼저 고른 것은 무엇이었나요? 왜 그것부터였나요?

**Q-E3.** 이 목록에 없는데 꼭 있었으면 하는 산출물이 있나요?
- 후속: 지금은 어떻게 만드시나요? (직접 작성 / 다른 도구 / 안 만듦)

**관찰 포인트**

- 미포함 산출물 수집

### 공통 설정

- 환경: 실제 Claude Code + VAIS Code 플러그인 (최신 버전)

### T1. Project Profile 합의 → 정책 게이트 작동 확인

**시나리오 설명**
"새로운 SaaS 피처를 만들려고 합니다. 오늘 처음 VAIS를 시작하는 것처럼 ideation부터 진행해주세요. 프로젝트는 B2B SaaS MVP, 유럽 사용자 포함, 구독 모델입니다."

**태스크 순서**
1. CEO에게 피처 설명 후 ideation 시작
2. ideation 종료 시 Project Profile 합의 화면 확인
3. CPO plan 단계에서 DPIA 산출물이 "Scope: 자동 포함" 표시 확인
4. AARRR이 "User-select" 표시로 사용자 선택 대기 확인

**성공 기준**
- Profile 합의 소요 시간 5분 이하
- DPIA 자동 포함 메시지를 사용자가 이해했다는 반응 (언어·행동)
- AARRR 선택 UX에서 혼란 없이 선택 완료

**실패 신호**
- Profile 12변수 중 3개 이상에서 "이게 무슨 뜻이죠?" 질문
- DPIA 자동 포함을 "왜 이게 갑자기 생겼지?"라고 의아해하는 반응
- 정책 게이트 화면을 건너뛰려는 시도

**측정 지표**
- Profile 합의 시간 (초)
- Profile 변수 질문 횟수
- 정책 분류 오해 발생 횟수

---

### T2. Deprecated Sub-agent 호출 → 새 에이전트 안내 확인

**시나리오 설명**
"기존에 만들던 프로젝트가 있습니다. COO에게 'release-engineer를 불러줘'라고 해주세요."

**태스크 순서**
1. COO에게 release-engineer 호출 요청
2. deprecate notice 화면 확인 ("release-engineer는 5개 sub-agent로 분해되었습니다")
3. 5개 후보 중 현재 필요한 것 선택 (시나리오: CI/CD 파이프라인 설정이 목적)
4. ci-cd-configurator 호출 및 Profile 기반 자동 게이트 통과 확인

**성공 기준**
- deprecate notice를 5분 내 이해하고 올바른 후속 에이전트 선택
- Profile 조건 기반 skip(예: "이미 GitHub Actions 설정 파일 있음 → skip") 메시지 이해
- 선택한 sub-agent의 산출물 정의를 사전에 이해하고 진행

**실패 신호**
- deprecate notice를 무시하고 강제로 release-engineer 호출 재시도
- 5개 후보 앞에서 30초 이상 망설임
- "왜 없어졌어요?"라는 불만 표출 후 중단

**측정 지표**
- deprecate notice 이해 소요 시간 (초)
- 후속 에이전트 선택 시간 (초)
- 선택 실수 횟수 (ci-cd가 아닌 다른 것 선택 후 수정)

### T3. User-select 산출물 다중 선택 → 자동 작성

**시나리오 설명**
"새 프로젝트의 시장 분석 단계입니다. CBO의 market-researcher를 불러주세요. market-researcher가 제안하는 분석 프레임워크 중 원하는 것을 선택해서 진행해주세요."

**태스크 순서**
1. CBO → market-researcher 호출
2. User-select 산출물 목록 표시 확인 (PEST, Five Forces, SWOT 등 4가지 옵션 제시)
3. 참여자 자유 선택 (제약 없음)
4. 선택된 산출물 자동 작성 시작 확인

**성공 기준**
- 4가지 옵션의 차이를 이해하고 선택 (단순 "첫 번째 것" 선택이 아닌 의도적 선택)
- 선택 완료 후 "왜 이것을 고르셨나요?" 질문에 이유 설명 가능
- 자동 작성 결과물을 보고 "이건 내가 원한 것"이라는 확인

**실패 신호**
- "뭐가 다른지 모르겠어요"라는 옵션 차이 혼란
- 아무거나 고르고 "나중에 보면 되죠"라는 무관심
- 작성 결과물이 선택한 프레임워크와 다른 구조로 나옴

**측정 지표**
- 옵션 이해 질문 횟수
- 선택 소요 시간 (초)
- 선택 의도 설명 성공 여부 (0/1)
- 결과물 만족도 (1~5점 척도)

## 5. 분석 프레임워크

### 5.1 Affinity Diagram 카테고리 후보

인터뷰 7개 세션 완료 후 affinity mapping 실시. 아래 8개 카테고리를 초기 버킷으로 사용하되 귀납적으로 재조정.

1. **산출물 과잉 통증** — 원하지 않는데 만들어진 경험, 불필요 산출물 예시
2. **산출물 부족 통증** — 없는데 필요한 산출물, 대체 수단
3. **역할 인지 혼란** — "이 sub-agent가 뭘 하는지 몰랐다" 경험
4. **Profile 변수 혼란** — 이해하기 어려운 변수, 누락 변수
5. **정책 분류 직관** — 카드 정렬에서 자신 있게 vs 망설이며 분류한 것
6. **위임 경계** — 믿고 그대로 쓰는 산출물 vs 반드시 검토하는 산출물
7. **단계 간 불일치** — Core→Why→What→How 사이 모순 경험
8. **우선순위 선호** — 가장 먼저 필요한 산출물, 없어도 되는 산출물

### 5.2 정량 메트릭 (5개)

| # | 메트릭 | 측정 방법 | 목표값 |
|---|--------|---------|-------|
| M-1 | Profile 합의 시간 | T1 시나리오 타이머 | 5분 이하 (300초) |
| M-2 | 정책 분류 직관성 | 카드 정렬 일치율 (정답 대비) | 7/10 이상 |
| M-3 | NPS (Net Promoter Score) | 인터뷰 마무리 0~10점 질문 | 현재 baseline 측정 (목표값 미설정) |
| M-4 | SUS (System Usability Scale) | 사용성 테스트 후 10문항 설문 | 68점 이상 (평균 이상) |
| M-5 | 산출물 신뢰도 | "이 산출물을 그대로 쓰겠다" 비율 | 상위 5개 산출물 기준 70% 이상 |

**M-3 측정 질문**: "주변 동료나 개발자에게 VAIS를 추천할 의향이 어느 정도인가요? 0~10점으로 말씀해주세요."

**M-4 SUS 10문항** (기존 표준 SUS 그대로 적용):
1. 이 시스템을 자주 사용하고 싶다.
2. 이 시스템은 불필요하게 복잡하다.
3. 이 시스템은 사용하기 쉽다.
4. 이 시스템을 사용하려면 전문가의 도움이 필요할 것 같다.
5. 이 시스템의 여러 기능이 잘 통합되어 있다.
6. 이 시스템에 일관성이 없다.
7. 대부분의 사람들이 이 시스템을 빠르게 배울 것이다.
8. 이 시스템은 사용하기 번거롭다.
9. 이 시스템을 사용할 때 자신감을 느낀다.
10. 이 시스템을 사용하기 전에 많은 것을 배워야 했다.

## 6. Persona 심화 (P1 / P2 / P3)

product-researcher의 기본 페르소나(P1 솔로 빌더, P2 스타트업 CTO, P3 엔터프라이즈 PM)를 UX 관점으로 확장한다.

---

### P1. 솔로 빌더 — "내가 모르는 영역"을 어떻게 위임하는가

**기본 프로필**
- 이름(가명): 권현준 (30대 초반)
- 역할: 1인 개발자, SaaS 또는 Claude Marketplace 플러그인 제작
- 기술 배경: 풀스택 개발 가능, 기획/마케팅/보안은 약함
- VAIS 사용 기간: 1~3개월

**행동 변수 스펙트럼**

| 변수 | 스펙트럼 | P1 위치 |
|------|---------|---------|
| 위임 의향 | 모두 직접 ◄────────► 모두 위임 | 기술: 직접 / 기획·보안: 위임 |
| 도구 학습 투자 | 최소 ◄────────► 최대 | 중간 (필요할 때만 학습) |
| 결과물 신뢰 | 의심 ◄────────► 신뢰 | 코드는 의심 / 문서는 신뢰 |
| 프로세스 준수 | 즉흥 ◄────────► 엄격 | 즉흥 쪽 (빠르게 반복) |

**Mental Model: "내가 모르는 영역" 위임 방식**

P1은 모르는 영역(예: STRIDE 위협 모델, DPIA, GTM 전략)에 대해 "이 산출물이 맞는지 판단할 기준이 없다"는 인식을 가진다. 그래서:

1. "일단 만들어달라" → 결과물을 보고 "이게 맞는 말이네" 또는 "잘 모르겠다"로 분류
2. 모르는 산출물은 그대로 파일로 저장하되 실제로 읽지 않는 경향
3. "나중에 필요하면 참고하겠다"는 안심 효과로 산출물 존재 자체를 가치로 인식

**UX 설계 함의**
- 모르는 영역 산출물일수록 "이 산출물이 필요한 이유" 한 줄 설명이 중요
- "당신의 프로젝트에 이 산출물이 왜 필요한지" = Profile 기반 맥락 설명
- P1에게 50개 카탈로그는 압도적 → 진입점을 Always(17개) 기준으로 좁혀 표시 필요

**감정 여정 (P1)**

| 단계 | 감정 | 행동 | Pain Point |
|------|------|------|----------|
| VAIS 첫 실행 | 기대 + 불안 | CEO에게 아이디어 설명 | "어디서 시작해야 하지?" |
| ideation | 흥미로움 | Q&A 대화 진행 | "질문이 너무 많다" |
| Profile 합의 | 약간의 부담 | 12변수 체크 | "이 변수가 뭔지 모르겠다" |
| 산출물 자동 생성 | 뿌듯함 | 결과물 확인 | "왜 이게 만들어졌지?" |
| 불필요 산출물 발견 | 혼란 + 약간의 짜증 | 삭제 또는 무시 | "내가 원한 게 아닌데" |
| 작업 완료 | 만족 + 약간의 불안 | 배포 진행 | "보안 검토 충분히 됐나?" |

### P2. 스타트업 CTO — 어디서 confirm 받고 어디서 위임하는가

**기본 프로필**
- 이름(가명): 이수빈 (30대 중후반)
- 역할: 시리즈 A 스타트업 CTO, 3~8인 팀 관리
- 기술 배경: 백엔드/인프라 강함, 비즈니스/법무 약함
- VAIS 사용 기간: 2~4개월, 팀 전체 도입 검토 중

**행동 변수 스펙트럼**

| 변수 | 스펙트럼 | P2 위치 |
|------|---------|---------|
| 위임 의향 | 모두 직접 ◄────────► 모두 위임 | 설계 결정: 직접 / 문서 작성: 위임 |
| 리스크 감수 | 매우 조심 ◄────────► 과감 | 법무·보안: 조심 / 기술 실험: 과감 |
| 결과물 신뢰 | 의심 ◄────────► 신뢰 | 설계 결정은 반드시 검토 |
| 팀 전파 의향 | 없음 ◄────────► 강함 | 중간 (검증 후 전파) |

**Mental Model: 의사결정 위임 구조**

P2는 의사결정을 두 층으로 분리한다:

- **레이어 1 (전략·설계)**: CEO/CPO/CSO 산출물 → 반드시 본인이 검토, 수정 후 승인
- **레이어 2 (실행·문서)**: CTO 하위 sub-agent 산출물 → 검토 없이 신뢰하거나 팀원에게 위임

"ADR은 내가 직접 쓰거나 최소한 검토해야 한다. API Spec은 backend-engineer가 만든 걸 QA 엔지니어가 검증하면 된다."

**Confirm 받고 싶은 영역**
- 아키텍처 결정 (ADR)
- 보안 위협 모델 (STRIDE)
- 가격 정책 (Pricing Tier Canvas)
- 법무 관련 (DPIA, Terms of Service)

**위임하고 싶은 영역**
- 시장 분석 (PEST, Five Forces)
- 기술 문서 (API Spec, ER Diagram)
- 운영 문서 (Runbook, SLO)
- 마케팅 카피

**UX 설계 함의**
- "이 산출물은 CTO 검토 권장" 레이블이 있다면 P2의 신뢰 상승
- Profile에 "팀 사용 여부" 변수 추가 시 위임 경계 자동 조정 가능
- 산출물별 "review_recommended: bool" 메타 필드가 P2에게 핵심 기능

**감정 여정 (P2)**

| 단계 | 감정 | 행동 | Pain Point |
|------|------|------|----------|
| 신규 피처 시작 | 집중 + 시간 압박 | CPO plan 요청 | "PRD가 얼마나 걸리지?" |
| 산출물 검토 | 비판적 | 섹션별 검토 + 주석 | "이 내용 틀렸는데" |
| 팀 공유 | 불안 | Slack에 링크 공유 | "팀원이 이걸 이해할까?" |
| 보안 리뷰 | 긴장 | CSO 호출 | "OWASP 몇 개나 걸릴까?" |
| 배포 결정 | 책임감 | COO 점검 후 최종 승인 | "내가 놓친 것 없나?" |

### P3. 엔터프라이즈 PM — 어느 산출물이 필수인가

**기본 프로필**
- 이름(가명): 박지현 (30대 후반~40대 초반)
- 역할: 대기업 또는 금융권 IT 부서 PM/PO
- 기술 배경: 비개발자, 요구사항/프로세스 중심
- VAIS 사용 기간: 3~6개월, 컴플라이언스 관련 산출물 중심 사용

**행동 변수 스펙트럼**

| 변수 | 스펙트럼 | P3 위치 |
|------|---------|---------|
| 컴플라이언스 중요도 | 선택 ◄────────► 필수 | 매우 높음 (필수 쪽 극단) |
| 변화 수용 | 저항 ◄────────► 적극 | 중간 (검증된 것만 수용) |
| 문서 완성도 요구 | 초안 OK ◄────────► 완성본 필요 | 외부 제출용: 완성본 필요 |
| AI 신뢰 | 불신 ◄────────► 신뢰 | 낮음 (모든 산출물 검토) |

**컴플라이언스 흐름: 필수 산출물 매핑**

P3 관점의 "반드시 있어야 하는" 산출물:

| 카테고리 | 필수 산출물 | 이유 |
|---------|-----------|------|
| 법무·개인정보 | DPIA, Privacy Policy 초안 | GDPR/개인정보보호법 |
| 정보보안 | STRIDE Threat Model, OWASP ASVS | 내부 보안 감사 |
| 품질 | Test Plan, Test Case Matrix | QA 게이트 통과 |
| 운영 | Runbook, SLO Definition | 서비스 수준 협약 |
| 변경 관리 | ADR, RFC/TDD | 변경 이력 추적 |

**"없어도 되는" 산출물 (P3 관점)**
- Lean Canvas (내부 서비스 — BM 검토 불필요)
- SEO Audit (내부망 서비스)
- AARRR (사용자 획득 메트릭 — 내부 서비스 해당 없음)
- Marketing Funnel (마케팅 없음)

**UX 설계 함의**
- Profile `geography: regulated-region` + `handles_pii: true` → DPIA 자동 Always 격상
- `deployment.target: on-prem` → Cloud Cost, FinOps 자동 skip
- "규제 기관 제출용" 플래그 옵션 → 산출물 완성도 기준 강화 모드

**감정 여정 (P3)**

| 단계 | 감정 | 행동 | Pain Point |
|------|------|------|----------|
| 프로젝트 시작 | 부담감 | 요구사항 정리 | "DPIA 언제 시작하지?" |
| VAIS 도입 검토 | 의심 | 컴플라이언스 산출물 확인 | "이 AI가 만든 거 쓸 수 있나?" |
| DPIA 생성 | 조심 | 세부 내용 직접 검토 | "법적 책임은 내가 지는데" |
| 내부 감사 준비 | 긴장 | 산출물 전체 점검 | "누락된 게 없나?" |
| 결과 제출 | 안도 (부분적) | 승인 대기 | "AI가 만든 걸 감사가 인정할까?" |

## 7. Empathy Map (3 페르소나)

### Empathy Map: P1 — 솔로 빌더

**Says (말하는 것)**
- "일단 만들어봐. 나중에 고치면 되지."
- "이 산출물 뭔지 잘 모르겠는데 일단 파일로 저장해둬."
- "보안은 나중에 신경 써도 되겠지."
- "너무 많이 만들어주는 것 같은데?"

**Thinks (생각하는 것)**
- "내가 모르는 걸 대신 해줘서 고마운데, 이게 맞는 건지 어떻게 알지?"
- "이 산출물이 실제로 필요한 건지 나중에 알게 되겠지."
- "만들어준다고 해서 다 써야 하는 건 아니잖아."
- "이거 다 읽으려면 시간이 얼마나 걸리지?"

**Does (하는 것)**
- VAIS를 '빠른 초안 생성 도구'로 사용
- 불필요한 산출물은 삭제하지 않고 방치
- 기술 산출물(API Spec, ERD)은 직접 검토, 비기술 산출물은 그대로 저장
- 하나의 피처에 여러 C-Level 호출보다 CTO 위주 사용

**Feels (느끼는 것)**
- 혼자서 모든 역할을 해야 한다는 압박감
- AI가 대신 해줄 때의 해방감
- 내가 이해 못하는 산출물이 쌓일 때의 불안감
- "이 정도면 됐다"고 넘어가는 타협 감정

## 8. Handoff 메모 — C-Level 큐레이션을 위한 우선순위

다음 항목은 design phase C-Level 큐레이션 시 우선 반영을 권장한다.

### 즉시 설계 반영 후보

1. **Profile 변수 설명 강화** — 각 변수에 "예: 유럽 사용자가 있다면 geography: global" 한 줄 예시 추가 → P1의 "뭔지 모르겠다" 통증 직접 해소
2. **산출물 맥락 설명** — 각 산출물에 "당신의 프로젝트에 이것이 필요한 이유" 한 줄 → Profile 연계 표시
3. **review_recommended 메타필드** — P2 CTO의 "내가 봐야 하는 것" 식별 지원

### 인터뷰 전 설계 불확정 영역 (인터뷰 결과 후 결정)

- Profile 12변수 중 추가/제거 항목 → H-1 검증 후
- 정책 4분류 레이블명 → H-2 카드 정렬 결과 후
- User-select UX 최적 패턴 → T3 사용성 테스트 후
