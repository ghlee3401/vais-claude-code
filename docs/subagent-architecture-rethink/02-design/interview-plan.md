# Topic: 외부 인터뷰 + Usability Test 계획 (CPO 큐레이션)

> Source: `_tmp/ux-researcher.md` (757 lines) + `_tmp/product-discoverer.md` 인터뷰 질문 5개 → CPO 큐레이션. SC-10(b) 이행을 위한 외부 검증 인터뷰 설계.

## 1. 인터뷰 가설 7개

| ID | 가설 | 검증 |
|----|------|------|
| **H-1** | Project Profile 12 변수가 실제 프로젝트 분류에 충분 | B 정책 산출물 자동 판단 적중률 |
| **H-2** | 정책 4분류(A/B/C/D)가 사용자 직관과 일치 | 카드 정렬 10개 중 7+ 일치 |
| **H-3** | "쓸데없이 만든다" 통증이 본인뿐 아니라 외부 보편적 (5 중 3+) | 직접 진술 빈도 |
| **H-4** | 50+ 카탈로그에서 사용자가 실제 원하는 우선순위가 식별 가능 | "5개만 고르면" 집계 |
| **H-5** | sub-agent 호출 전 산출물을 인지하고 있는가 | 호출 전 인식률 |
| **H-6** | 단계 간 alignment 누락(Core↔Why↔What↔How 모순)을 사용자가 경험 | 직접 사례 진술 |
| **H-7** | "이상적 위임 경계"의 공통 패턴 추출 가능 | 산출물 특성 기반 클러스터링 |

## 2. 인터뷰 스크립트 (45분 세션)

```
A. 도입 (5분)
   - 자기소개 / 현재 역할 / VAIS 사용 빈도

B. 사용 패턴 (10분)
   - 어떤 AI agent 도구 사용? VAIS는 얼마나?
   - 가장 자주 호출하는 sub-agent / C-Level?
   - 가장 만족 / 불만족 에이전트?

C. 통증 발굴 (15분) ← discoverer Q1~Q5 통합
   - "AI가 원하지 않는 것을 만든" 최근 경험 (Q1)
   - 프로젝트 특성에 따라 달랐으면 했던 산출물 (Q2)
   - "이 도구가 뭘 하는지 모르겠다" 순간 (Q3)
   - 산출물 포맷 일관성 문제 (Q4)
   - 가장 중요한데 미해결인 것 (Q5 — 신규 OPP 발굴)

D. 정책 + Profile 검증 (8분)
   - Profile 12 변수 카드 노출 → "이 변수만으로 내 프로젝트 분류 가능?"
   - 4분류 정책 카드 정렬 (10개 산출물 → A/B/C/D 분류)
   - "skip 메시지가 어떻게 보였으면?" (think-aloud)

E. 카탈로그 우선순위 (5분)
   - 50개 산출물 카드 → "5개만 고른다면?"
   - 누락된 산출물 자유 발화

F. 마무리 (2분)
   - 추가 의견 / 후속 인터뷰 동의
```

각 섹션의 후속 질문(3~5개)·관찰 포인트는 `_tmp/ux-researcher.md` 섹션 2 참조.

## 3. 모집 (5~7명, 다양성 매트릭스)

| 슬롯 | 프로젝트 유형 | 역할 | VAIS 경험 |
|:----:|------------|------|---------|
| 1 | SaaS B2B | 솔로 빌더 | 1주~1개월 |
| 2 | Internal tool | 스타트업 CTO | 1~3개월 |
| 3 | OSS plugin | 개인 컨트리뷰터 | 1주 미만 |
| 4 | API-only | Backend lead | 1~3개월 |

| 7 (예비) | Enterprise | 시니어 PM/아키텍트 | 가능 시 |

**스크리닝 3문항**:
1. VAIS Code를 실제 프로젝트에 적용한 경험이 있는가?
2. 최근 4주 내 sub-agent를 5회 이상 호출한 경험이 있는가?
3. 다른 AI agent 도구(ChatGPT/CrewAI/Devin 등)를 비교 사용한 경험이 있는가?

## 4. Usability Test 시나리오 (3개)

### T1. Profile 합의 → 정책 게이트 작동
- **시나리오**: 신규 피처 ideation → ideation 종료 시 Profile 12변수 합의 → 첫 sub-agent 호출 → 정책 B 산출물이 자동 skip 되는 상황을 사용자가 인지·이해
- **성공 기준**: skip 메시지를 본 사용자가 30초 내 "왜 skip 됐는지" 정확히 설명
- **측정 지표**: Profile 합의 시간 / skip 메시지 이해도 / 만족도

### T2. release-engineer deprecation 안내 → 새 sub-agent 선택
- **시나리오**: 기존 release-engineer 호출 → deprecation notice 출력 → 5개 신규 sub-agent 중 적절한 것 선택
- **성공 기준**: 사용자가 자기 프로젝트에 맞는 sub-agent를 1분 내 식별
- **측정 지표**: 식별 시간 / 선택 정확도 / 안내 명확성 5점

### T3. User-select 정책 다중 선택 → 자동 작성
- **시나리오**: 시장 분석에서 PEST/Five Forces/SWOT 다중 선택 → 자동 작성 → 결과 검토
- **성공 기준**: 다중 선택 UX 직관성 + 결과 산출물 신뢰도
- **측정 지표**: 선택 시간 / 신뢰도 NPS

## 5. 분석 프레임워크

### Affinity Diagram 카테고리 (8개)
1. Default-execute 통증 진술
2. Profile 변수 추가/제거 의견
3. 정책 4분류 직관성
4. 산출물 우선순위 패턴
5. sub-agent 정체성 인식
6. 단계 간 alignment 경험
7. 위임 경계의 공통 멘탈 모델
8. 신규 통증 (OPP-1~8 외)

### 정량 메트릭 5개
| 메트릭 | 측정 방법 |
|--------|---------|
| Profile 합의 시간 | T1 시작 ~ profile.yaml 생성까지 |
| 정책 카드 정렬 일치율 | 10개 중 정확히 분류한 개수 |
| NPS (Net Promoter Score) | "VAIS를 동료에게 추천?" 0~10 |
| SUS (System Usability Scale) | 표준 10문항 |
| 산출물 신뢰도 | "이 산출물을 외부에 그대로 제출 가능?" 5점 |

## 6. Persona 심화 (researcher의 P1/P2/P3 → UX 관점 확장)

### P1. 솔로 빌더 — 위임 멘탈 모델
- "내가 모르는 영역" → AI에게 위임. 단, 산출물이 외부 협업자(투자자/고객)에게 보일 수준이어야 함
- 행동 변수: 인터넷 검색 시간 vs AI 위임 시간 비율
- 감정 여정: 혼란(첫 호출) → 의심(첫 산출물) → 신뢰(정전 출처 확인) → 의존(반복 사용)

### P2. 스타트업 CTO — 2-레이어 위임
- 레이어 1: 자신이 결정 (전략·아키텍처)
- 레이어 2: 팀에게 위임 (CPO·CSO·COO 영역) — 팀 표준화 도구로 VAIS 활용
- 행동 변수: 산출물을 팀과 공유하는 빈도 (높을수록 template 표준화 가치 큼)
- 감정 여정: 효율성 기대 → 실망(첫 산출물 비표준) → 재신뢰(template 기반 산출) → 적극 사용

### P3. 엔터프라이즈 PM — 컴플라이언스 우선
- 산출물의 정전 출처가 감사 설득력
- 행동 변수: "이 결정의 근거가 무엇인가" 추적 빈도 (감사 시즌에 급증)
- 감정 여정: 회의(AI 결정 신뢰?) → 검토(_tmp/ 추적성 확인) → 채택(정전 출처) → 표준화(사내 도입)

## 7. Empathy Map (3 personas)

| | P1 솔로 빌더 | P2 스타트업 CTO | P3 엔터프라이즈 PM |
|---|-----------|----------------|------------------|
| **Says** | "이거 필요 없는데 왜 만들지?" | "팀에게 어떻게 공유?" | "감사 통과 가능?" |
| **Thinks** | "내가 모르는 영역도 채워줘" | "표준화가 더 중요" | "AI 결정 근거 어디?" |
| **Does** | 산출물 삭제·재작성 반복 | 직접 template 수정 | 정전 출처 확인 |
| **Feels** | 좌절(시간 낭비) → 신뢰(skip 작동) | 부담(표준 부재) → 안도(template) | 의심(자동화) → 신뢰(추적성) |

## 9. 작업 진행 우선순위 (인터뷰 전)

UX-researcher 결과에서 **인터뷰 결과 없이도 즉시 design/do에 반영 가능한 항목 3가지** 식별:

1. Profile 변수 schema에 각 변수당 한 줄 예시 추가 (인지부하 완화)
2. 산출물 카탈로그에 "이 프로젝트에 필요한 이유" 컨텍스트 컬럼 추가
3. Template metadata schema에 `review_recommended: bool` 신규 필드 (B 정책 보완)

→ do phase에서 즉시 적용. 인터뷰 결과 후 추가 보완은 별도 patch.
