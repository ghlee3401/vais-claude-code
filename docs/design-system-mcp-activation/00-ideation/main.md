## 요청 원문

> "현재 ui designer가 ui-ux-pro-max mcp를 항상 부르고 있나?"

→ fact-finding 결과 **한 번도 부르지 않음** (호출 권한 자체 없음, MCP 비활성). v0.48부터 v0.60.0까지 12 minor에 걸쳐 미활성 상태로 유지된 `vendor/ui-ux-pro-max` + `mcp/design-system-server.json` 설계도를 실제로 **활성화**하는 방향으로 ideation 진행.

---

## Key Points

- **현황 (fact-finding 결과)**:
  - `vais.config.json > mcp.enabled: false` (v0.48+ 부터 미활성)
  - `.mcp.json` 미존재, `.claude-plugin/plugin.json` 에 mcp 등록 없음
  - `agents/cto/ui-designer.md > tools` 에 `mcp__...` 권한 없음
  - `mcp/design-system-server.json` 스펙은 정의되어 있으나 등록되지 않음 (3개 tool: `design_search`, `design_system_generate`, `design_stack_search`)
  - 결과: ui-designer 가 design token 없이 디자인하거나, 사용자가 수동으로 `vendor/ui-ux-pro-max/scripts/search.py` 를 호출해야 함

- **v0.48 → v0.60.0 12 minor 미활성 사유**: blocker 없음. 단순 우선순위 — 다른 곳 (subagent-architecture-rethink 등) 이 바빠서 손이 못 갔음. 기술적/품질적 장애물 X.

- **활용 범위**: `mcp/design-system-server.json` 의 3개 tool 모두 사용 (`design_search` + `design_system_generate` + `design_stack_search`). 일부만 활성화하지 않음 — 원설계대로 풀 운용.

- **활성화 강도**: v0.60.0 Profile gate ON 결정과 같은 패턴 — OSS 사용자 default ON + 자동 호출. 다만 Profile gate 와 결정적으로 다른 점은 **Python3 + vendor 데이터 의존이 강제** 된다는 것.

- **실패 정책**: graceful degradation 이 아닌 Hard fail 선택. Python3 미설치 / `vendor/ui-ux-pro-max` 손상 시 design phase 중단 + 명확한 설치 가이드 표시. "조용히 토큰 없이 진행" 이 오히려 사용자 혼란을 키운다는 판단.

## Decisions

| # | Decision | 근거 | 영향 |
|:-:|----------|------|------|
| **D-1** | MCP 활성화 scope = OSS default ON (1인 사용 한정 X) | v0.60.0 Profile gate ON 패턴 답습. blocker 없으면 default 활용이 정석 | 모든 OSS 사용자 환경 영향 — Python3 + vendor 데이터 강제 |
| **D-2** | design phase 진입 시 `design_system_generate` 자동 호출 | 사용자가 매번 명시 트리거하는 UX 비용 회피. design = 토큰 필요 = 자동 생성이 정의 | design phase 첫 진입 시 MASTER.md 자동 영속화 |
| **D-3** | 3개 tool 모두 사용 (search + generate + stack-search) | `mcp/design-system-server.json` 원설계 의도 존중. 부분 활성화 = drift 유발 | ui-designer (search/generate) + frontend-engineer (stack-search) 권한 분배 필요 (plan 영역) |
| **D-4** | Python3 / vendor 미설치 시 Hard fail (graceful skip X) | 자동 호출 + 토큰 없이 진행 = 사일런트 품질 저하. 명시적 실패가 사용자에게 더 친절 | OSS 사용자 첫 진입 장벽 ↑. README + 설치 가이드 보강 필수 |

## Open Questions

(plan 단계에서 결정)

- **Q-1 (호출 책임)**: 어느 agent 가 어떤 tool 호출? 후보:
  - (a) ui-designer 가 search + generate 호출, frontend-engineer 가 stack-search
  - (b) CTO orchestrator 가 분기 + 각 agent 는 read 만
  - (c) 새 agent (design-token-generator) 신설
- **Q-2 (호출 시점)**: `design_stack_search` 는 design phase 인가 do phase 인가?
- **Q-3 (캐싱)**: `design-system/{feature}/MASTER.md` 가 이미 있으면 `design_system_generate` skip? 재생성 트리거 조건?
- **Q-4 (Python 버전)**: 최소 요구 버전 (3.8+? 3.10+? 3.11+?). `vendor/ui-ux-pro-max/scripts/search.py` 실측 의존 확인 필요.
- **Q-5 (vendor 무결성)**: `vendor/ui-ux-pro-max/data` 손상 / 누락 감지 메커니즘. checksum? 파일 수 검사?
- **Q-6 (마이그레이션)**: 기존 v0.60.0 사용자 (mcp.enabled=false) 자동 true 전환할지, 명시 opt-in 으로 둘지. Profile gate 와 동일하게 fallback default = true 로 갈지.
- **Q-7 (Hard fail UX)**: 실패 메시지 본문 + 설치 가이드 링크 (README / docs / 외부 URL). i18n 한국어/영어?
- **Q-8 (테스트)**: MCP 통합 테스트 전략. mock subprocess vs 실제 Python 호출. CI 환경 Python3 설치?
- **Q-9 (`description` 문구 정정)**: ui-designer.md 의 "Consumes design tokens generated by UI/UX Pro Max" 표현이 호출 관계로 오해됨. 활성화 후엔 사실이 되지만 호출 주체 명시 필요.

## Raw Context (optional)

### CEO 첫 진단 발췌

> "v0.48+ 활용 예정" 약속이 v0.60.0까지 12 minor 미루어짐. ui-designer.md description 의 "Consumes design tokens generated by UI/UX Pro Max" 는 호출 관계가 아니라 파일 소비 관계 — 누군가가 사전에 design-system/{feature}/MASTER.md 를 만들어둠 → ui-designer 는 Read 로 읽기만 함. 현재는 그 "누군가" 의 자동화도 끊겨 있음.

### 분기 결정 4가지 흐름

1. Why blocked → "우선순위" (blocker 없음 → 진행 정당화)
2. Core value → "3개 모두" (완전 활성화 의지 확인)
3. Scope × Trigger → "A. OSS default ON + 자동 호출" (Profile gate 패턴 답습)
4. Fallback → "Hard fail" (사일런트 실패 회피, 명시적 가이드 우선)
