1. Claude Code 하네스 재료

왜 '하네스'라고 부르나?

Claude 같은 대형 언어 모델은 일반 목적 엔진입니다. 똑똑하지만 여러분의 프로젝트가 어떤 규칙을 쓰는지, 어떤 파일 구조를 쓰는지, 어떤 실수를 절대 하면 안 되는지 전혀 모릅니다. 그래서 모델에게 매번 "이 프로젝트는 이러이러하고…" 를 처음부터 설명하면 토큰만 잡아먹고 일관성도 떨어집니다.

하네스(harness)는 안전벨트 + 훈련복 + 지시서 한 벌입니다. 모델에게:

  • 컨텍스트를 자동으로 주입해서 똑같은 설명을 반복하지 않게 만들고,
  • 제약을 걸어서 위험한 행동(rm -rf /, 민감정보 커밋 등)을 막고,
  • 도구를 골라서 쥐어주고,
  • 상태를 파일로 남겨서 다음 세션에도 이어지게 합니다.

Claude Code는 이 네 가지를 만들 수 있는 6개의 기본 재료를 제공합니다. 하나씩 보겠습니다.

1.1 CLAUDE.md — 프로젝트 설명서

Claude Code가 세션을 시작할 때 자동으로 읽는 마크다운 파일입니다. ./CLAUDE.md, ./.claude/CLAUDE.md, ~/.claude/CLAUDE.md 순서로 계층적으로 병합되어 모든 대화의 시스템 프롬프트에 주입됩니다.

📌 핵심 용도
"이 프로젝트에서 반드시 지켜야 할 규칙"을 적어두는 곳. 코딩 스타일, 파일 경로 컨벤션, 빌드 명령어, 배포 절차, 금지 사항 등.

예시: VAIS Code 프로젝트의 CLAUDE.md 일부 —

markdown · CLAUDE.md 
# VAIS Code - Claude Code Plugin

## Mandatory Rules

1. **기획 없이 코드 금지** — `docs/{feature}/01-plan/` 기획서가 없으면 구현하지 않는다
2. **워크플로우 순서 준수** — plan → design → do → qa → report 순서 스킵 금지
3. **산출물 경로** — `docs/{feature}/{NN-phase}/main.md` 형식 준수
5. **위험 명령 금지** — `rm -rf`, `DROP TABLE`, `git push --force` 사용 금지
...

Claude는 코드를 작성하기 전에 이 규칙을 읽고 있으므로, 사용자가 매번 "산출물은 docs/… 에 넣어줘"라고 말하지 않아도 됩니다.

⚠️ 흔한 실수
CLAUDE.md를 "할 일 목록"이나 "일기장"처럼 쓰지 마세요. 이건 불변 규칙을 적는 곳입니다. 일회성 작업 설명은 대화 중에 하거나, 진행 중인 작업 맥락은 Memory에 남기세요(아래 1.2).

1.2 Memory 시스템 — 대화를 뛰어넘는 기억

CLAUDE.md가 "프로젝트 전체에 통하는 규칙"이라면, Memory는 "이 사용자와 일하면서 배운 것"을 기록하는 개인 서랍입니다. Claude Code는 각 프로젝트마다 ~/.claude/projects/<project-hash>/memory/ 폴더에 마크다운 파일로 저장합니다.

메모리는 네 가지 종류로 나뉩니다:

종류무엇을 저장예시
user사용자 역할·전문성·선호"이 사용자는 10년차 Go 개발자, React는 처음"
feedback고치라고/유지하라고 한 지시"요약 블록 뒤에 꼬리말 붙이지 말 것"
project진행 중인 일·목표·데드라인"2026-05-15 마켓플레이스 공개 목표"
reference외부 자원 포인터"장애는 Sentry 프로젝트 acme-prod에서 추적"

각 메모리는 프론트매터 + 본문으로 구성된 독립 파일이고, 인덱스는 MEMORY.md에 한 줄로 모입니다.

markdown · feedback_testing.md 
---
name: Integration tests must hit real DB
description: DB 통합 테스트는 mock 금지 — 프로덕션 마이그레이션 사고 이력
type: feedback
---

DB 통합 테스트는 실제 DB에 붙여야 한다.

**Why:** 지난 분기 mock 테스트는 통과했는데 prod 마이그레이션이 실패해 롤백함.
**How to apply:** integration_test/*.js 파일에서 DB 관련 mock 금지.
💡 언제 저장하나?
  • 사용자가 명시적으로 요청: "이거 기억해줘" → 즉시 저장
  • 수정 피드백: "그 방식 말고 이렇게" → why와 함께 저장
  • 비자명한 칭찬: "그래 그게 맞았어" → 판단 근거로 남김 (놓치기 쉬움)
  • 저장하지 말 것: 코드/git history로 알 수 있는 것, 진행 중 작업 상태

1.3 Hooks — 이벤트 기반 훅 시스템

Hooks는 "어떤 이벤트가 발생하면 내 스크립트를 실행하라" 고 Claude Code에게 지시하는 장치입니다. .claude/settings.json이나 플러그인의 hooks/hooks.json에 정의하고, 쉘 명령을 실행해 Claude의 컨텍스트에 정보를 주입하거나 행동을 막을 수 있습니다.

주요 이벤트

이벤트발생 시점활용 예
SessionStart새 세션 시작프로젝트 상태·버전·가이드 자동 주입
UserPromptSubmit사용자가 메시지 전송 직전프롬프트 전처리·금칙어 차단
PreToolUse도구 호출 직전위험 명령 차단 (rm -rf 등)
PostToolUse도구 호출 직후파일 변경 추적·자동 포맷
SubagentStart / SubagentStop서브에이전트 전환컨텍스트 격리·결과 집계
Stop응답 종료세션 요약·다음 액션 제안

VAIS Code의 hooks.json (발췌)

json · hooks/hooks.json 
{
  "hooks": {
    "SessionStart": [{
      "hooks": [{
        "type": "command",
        "command": "${CLAUDE_PLUGIN_ROOT}/hooks/run-node.sh ${CLAUDE_PLUGIN_ROOT}/hooks/session-start.js",
        "timeout": 5000
      }]
    }],
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "${CLAUDE_PLUGIN_ROOT}/hooks/run-node.sh ${CLAUDE_PLUGIN_ROOT}/scripts/bash-guard.js",
        "timeout": 3000
      }]
    }],
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "command": "... scripts/doc-tracker.js", "timeout": 5000 }]
    }]
  }
}

이 설정의 의미:

  • 세션 시작: session-start.js가 실행되어 VAIS 버전·사용법 안내를 모델 컨텍스트에 주입합니다. (이 가이드 맨 위에 보이는 "VAIS Code v0.58.0" 배너가 그 결과물)
  • Bash 실행 직전: bash-guard.js가 명령을 검사해 위험 명령이면 차단합니다.
  • Write/Edit 직후: doc-tracker.js가 어떤 문서가 바뀌었는지 .vais/ 상태에 기록합니다.
🚨 훅은 쉘 명령이라는 점을 잊지 마세요
훅은 모델이 직접 실행하는 코드가 아니라 Claude Code 런타임이 실행하는 외부 프로세스입니다. stdout으로 출력한 내용이 모델에게 보이고, 종료 코드로 차단/승인을 결정합니다. 그래서 훅은 "모델이 무시할 수 없는 하드 제약" 역할을 합니다.

1.4 Advisor Tool — 실행 중 가드레일

Advisor Tool은 Claude가 행동하기 전에 "이거 해도 되는지 먼저 물어보는" 감독자를 붙이는 기능입니다. Hooks가 "이벤트 발생 → 스크립트 실행" 의 비동기 반응이라면, Advisor는 모델의 의사결정 루프 안쪽에 사전 자문(pre-advisory)을 꽂는 장치에 가깝습니다.

동작 방식: 모델이 도구를 호출하기 전에 advisor가 정의된 규칙/패턴을 검사하고, 의심스러우면 사용자에게 질문을 던지거나 행동을 우회하라고 신호합니다. VAIS Code는 agents/_shared/advisor-guard.md로 기본 advisor 블록을 모든 C-Level 에이전트에 주입하고 있습니다.

Hooks vs Advisor 비교

특성HooksAdvisor Tool
실행 주체런타임 (외부 프로세스)모델 (내부 판단)
차단 강도강함 — 종료 코드로 거부보통 — 모델 선택 유도
활용보안·로깅·상태 기록정책 안내·질문 유도·사용자 체크포인트
비유공항 보안검색대옆에 앉은 멘토
🧭 조합 패턴
실전에서는 둘을 함께 씁니다. 절대 막아야 하는 건 Hook(예: rm -rf /), 사용자 판단이 필요한 건 Advisor(예: "이 규모의 리팩터를 한 PR에 넣을까요?").

1.5 Skills — 재사용 가능한 명령서

Skill은 "사용자가 /<skill-name>을 입력하면 이 지시서를 읽고 실행해" 라는 진입점 + 지시서의 묶음입니다. 플러그인이나 사용자 설정에 정의할 수 있습니다.

각 skill은 skills/<name>/SKILL.md 구조를 가지며, 프론트매터에 메타데이터 + 본문에 실행 지침을 담습니다:

markdown · skills/vais/SKILL.md (축약) 
---
name: vais
description: Orchestrates a virtual C-Suite (CEO, CPO, CTO, CSO, CBO, COO)
  to manage product strategy, technical implementation, security review...
allowed-tools: [Read, Write, Edit, Bash, Agent, TodoWrite, AskUserQuestion]
---

# VAIS Code — {action}

## 액션 목록
| 액션 | 설명 |
|------|------|
| `ceo [phase] [feature]` | CEO — 비즈니스 전략 + 동적 라우팅 |
| `cpo [phase] [feature]` | CPO — 제품 방향 + PRD |
...

## 실행 지침
1. Read 도구로 `skills/vais/phases/{c-level}.md` 파일을 읽으세요.
2. 그 안의 지침에 따라 실행하세요.

핵심은 Progressive Disclosure입니다. SKILL.md 자체는 짧게 유지하고, 실제 방법론은 필요할 때만 읽도록 phases/, utils/ 하위로 분산시킵니다. 토큰을 아끼면서도 풍부한 기능을 담을 수 있는 핵심 설계 패턴입니다.

💡 Skill과 Hook의 차이
  • Hook: 이벤트 기반, 자동 실행, 런타임이 실행
  • Skill: 명시적 호출, 사용자가 /이름 입력, 모델이 읽고 실행
Hook은 "자동차 브레이크", Skill은 "자동차 내비게이션 명령" 정도로 생각하면 됩니다.

1.6 Sub-Agents — 격리된 문맥 창 핵심

메인 세션 안에서 별도 컨텍스트 창을 가진 작은 Claude를 띄우는 기능입니다. 메인 Claude가 Agent 도구를 호출해 sub-agent를 시작하면, sub-agent는 자기만의 도구 목록 + 자기만의 지시서로 독립 실행하고, 결과 요약만 메인으로 돌아옵니다.

왜 중요한가?

🧠 컨텍스트 절약
리서치 에이전트가 파일 50개를 읽어도 요약 500자만 메인으로 리턴. 메인 문맥 창이 깨끗함.
🎯 전문성 격리
security-auditor는 OWASP 체크리스트만 읽고, frontend-engineer는 React 문서만 읽음.
⚡ 병렬 실행
frontend + backend + test 엔지니어를 동시에 실행해서 시간 단축.
🔒 권한 분리
research 에이전트는 Write 권한 제거 → 조사만 가능, 수정 불가.

Sub-agent 정의는 agents/<c-level>/<name>.md 파일에 프론트매터로 선언합니다:

markdown · agents/cto/backend-engineer.md 
---
name: backend-engineer
description: Implements backend APIs, server logic, and database integrations.
  Use when delegated by CTO for API implementation or server-side development.
tools: Read, Write, Edit, Glob, Grep, Bash, TodoWrite
---

# Backend Engineer

당신은 백엔드 엔지니어 서브에이전트입니다.
CTO로부터 위임받은 API 구현 작업만 수행합니다.
...
⚠️ 오해 방지
Sub-agent의 description 필드는 단순 주석이 아니라 메인 모델이 읽는 라우팅 힌트입니다. "누가 이 일을 해야 하는가"를 결정할 때 이 설명을 보고 고릅니다. 따라서 3인칭 + what + when으로 명확히 써야 합니다.

1.7 Output Styles · Settings · Plugin

Output Styles

응답의 톤과 포맷을 통일하는 마크다운. 모든 응답 앞뒤에 특정 포맷을 강제할 때 사용합니다. VAIS Code는 output-styles/vais-default.md에서 "하단 리포트 박스"를 강제하고 있습니다.

Settings

.claude/settings.json(프로젝트) / ~/.claude/settings.json(사용자)에서 권한, 환경변수, 모델, 테마, 훅을 설정합니다. 권한 예시:

json · .claude/settings.json 
{
  "permissions": {
    "allow": ["Bash(npm *)", "Bash(git log)", "Bash(git status)"],
    "deny":  ["Bash(rm -rf *)", "Bash(git push --force*)"]
  }
}

Plugin

위 재료들을 하나로 묶어 마켓플레이스에 배포 가능한 단위로 만든 것이 Plugin입니다. .claude-plugin/plugin.json에 스킬/에이전트/훅/output-styles/settings 진입점을 선언합니다. VAIS Code 자체가 Plugin으로 배포됩니다.

2. Harness Engineering 이란?

2.1 정의

Harness Engineering ≝ 범용 LLM을 특정 프로젝트·팀·도메인에 길들이기 위해 컨텍스트·제약·도구·상태 네 축을 설계하고 파일·훅·에이전트로 구현하는 실천.

비유하자면, 모델은 F1 엔진이고 하네스는 섀시·ECU·드라이버 지시서 한 벌입니다. 같은 엔진도 섀시에 따라 전혀 다른 차가 됩니다. 같은 Claude도 하네스에 따라 "코드 보조 도구"가 되거나 "C-Suite 시뮬레이션 조직"이 됩니다.

① Context — 뭘 알려주나?

  • 프로젝트 규칙 (CLAUDE.md)
  • 사용자/세션 기억 (Memory)
  • 실시간 상태 (SessionStart hook으로 주입하는 git status, 빌드 상태 등)

② Constraint — 뭘 막고/유도하나?

  • 기본 도구: Read/Write/Edit/Bash/Grep…

  • 파생 능력: Sub-agent(다른 '작은 Claude'도 도구)

④ State — 뭘 남기나?

  • 대화 너머까지: Memory 파일, 프로젝트 문서(docs/)
  • 진행 상태: .vais/status.json, .vais/agent-state.json
  • 감사 추적: 이벤트 로그, 훅 로그, git history

2.4 최소 → 고급 하네스 진화

레벨별로 어디까지 갈 수 있는지 감을 잡기 위한 스펙트럼입니다:

Level 0 · 맨몸

Level 3 · +Hooks

Level 5 · +Memory/Advisor
세션을 넘나드는 학습 + 사용자 체크포인트. 장기 운영 가능.
Level 6 · +MCP/외부 시스템
GitHub·Sentry·DB 연동. 실무 운영 레벨.
Level 7 · 플러그인
재배포 가능한 "하네스 패키지". VAIS Code가 여기.

3.2 C-Suite 계층 — 6 + 38

VAIS는 모델을 두 계층으로 나눕니다:

계층모델역할구성원
전략 레이어 Opus (큰 모델) 판단·라우팅·합성 CEO / CPO / CTO / CSO / CBO / COO (6명)
실행 레이어 Sonnet (작은 모델) 파일 생성·리서치·검증 38개 sub-agent (product-discoverer, backend-engineer, security-auditor, …)
💡 왜 두 계층인가?
Opus는 똑똑하지만 비싸고 느립니다. 결정이 필요한 곳(전략/합성)에만 Opus를 쓰고, 반복 작업(코드 생성/검색)은 Sonnet이 병렬로 처리하면 비용·속도·품질의 균형이 잡힙니다. 인간 조직의 임원-실무자 분리와 동형.

C-Level별 조직도 — 누가 누구를 거느리는가

왼쪽이 C-Level(전략, Opus), 오른쪽이 그 휘하 sub-agent(실행, Sonnet)입니다. 각 sub-agent 파일은 agents/{c-level}/{sub-agent}.md 경로에 있습니다.

3.4 문서 표준 — Rule 3 / 14 / 15 v0.57~58

VAIS의 가장 정교한 부분입니다. 여러 C-Level이 같은 피처를 건드리는 상황에서 race condition과 문서 오염을 막기 위한 3단계 규칙.

Rule 3 · 산출물 경로

모든 산출물은 docs/{feature}/{NN-phase}/main.md 형식. Phase↔Folder 매핑은 고정:

path mapping 
ideation → 00-ideation
plan     → 01-plan
design   → 02-design
do       → 03-do
qa       → 04-qa
report   → 05-report
🚫 레거시 경로 금지 (Rule 13)
docs/01-plan/ 같은 top-level NN- 패턴은 금지. 반드시 피처 이름이 먼저 오는 docs/{feature}/01-plan/ 구조. pre-commit hook이 자동 차단합니다.

Rule 14 · Sub-doc 보존 v0.57+

이건 VAIS만의 독특한 설계입니다. 흐름:

  1. Sub-agent는 축약 없이 전체 분석을 docs/{feature}/{NN-phase}/_tmp/{agent-slug}.md scratchpad에 기록
  2. C-Level은 _tmp/*.md를 모두 읽고 큐레이션 (채택/거절/병합)
  3. 결과를 topic별 {topic}.md 여러 개로 재구성 + main.md는 인덱스 역할
  4. _tmp/는 영구 보존 + git 커밋 (추적성)
💡 왜 이렇게 복잡하게?
Sub-agent 5명이 병렬로 main.md를 동시에 편집하면 race. 그렇다고 각자 결과를 축약해서 요약만 넘기면 원본 추론 과정이 날아감.

"원본은 _tmp/에 불변 기록하고, 합성은 C-Level이 순차적으로" 하는 설계로 병렬성·추적성·무결성을 모두 잡았습니다.

Rule 15 · C-Level 공존 v0.58+

여러 C-Level이 같은 main.md에 기여할 때의 규칙:

  • 진입 시 기존 main.md를 Read 필수
  • ## [CPO] …, ## [CTO] … 같은 H2 섹션으로 append만 (대문자)
  • 다른 C-Level의 섹션 수정·삭제 금지
  • topic 문서는 frontmatter에 owner: cpo 같이 소유권 명시
  • main.md가 200줄 초과 시 → topic 분리 필수 (size budget)
  • 위반 탐지: scripts/doc-validator.js의 W-OWN-*, W-MRG-*, W-MAIN-SIZE 경고

doc-validator.js

.hooks/pre-commit

3.7 vais.config.json — Single Source of Truth

모든 설정은 vais.config.json에 모입니다. 다른 파일에 하드코딩된 값이 있으면 리팩터링 대상으로 봅니다. 주요 키:

json · vais.config.json (발췌) 
{
  "version": "0.58.0",
  "workflow": {
    "phases": ["ideation", "plan", "design", "do", "qa", "report"],
    "mandatory": ["plan", "design", "do", "qa"],
    "docPaths": {
      "ideation": "docs/{feature}/00-ideation/main.md",
      "plan":     "docs/{feature}/01-plan/main.md",
      ...
    },
    "topicPresets": { "plan": ["requirements", "architecture", "threats"], ... },
    "subDocPolicy": { "enforcement": "warn", "tmpPreserve": true },
    "cLevelCoexistencePolicy": {
      "enforcement": "warn",
      "mainMdMaxLines": 200
    }
  },
  "cSuite": {
    "launchPipeline": ["cbo", "cpo", "cto", "cso", "cbo-gtm", "coo"],
    "dependencies": { "cto": [], "cso": ["cto"], "coo": ["cto"] }
  },
  "gates": { "gapThreshold": 0.9 }
}
💡 왜 SSoT가 중요한가?
CEO가 의존성을 판단할 때, CSO가 게이트 기준을 볼 때, pre-commit 훅이 경로 패턴을 검사할 때 — 모두 같은 파일을 읽습니다. 한 곳만 바꾸면 끝. 하네스 유지보수의 핵심 원칙입니다.

🔌 MCP (Model Context Protocol) 서버 활성화

mcp/ 디렉토리는 존재하지만 현재 비어있음. v0.48+ 활용 예정. 활용 가능 영역:

  • GitHub 연동 — issue/PR 자동 추적, CEO가 백로그 동기화
  • Linear/Jira 연동 — product-researcher가 실제 티켓 조회
  • Sentry/Datadog — incident-responder가 실시간 에러 조회
  • DB 직접 쿼리 — db-architect가 스키마 실측

🧠 Memory 시스템 통합

현재 Claude Code 기본 auto-memory는 개별 사용자 메모리로 동작. VAIS는 프로젝트 레벨 메모리를 아직 직접 관리하지 않습니다. 아이디어:

  • CEO가 피처 완료 시 "이 프로젝트에서 배운 패턴"을 자동 메모리화
  • CPO의 user interview 인사이트를 memory/reference로 저장 → 다음 피처 기획 시 활용
  • CTO incident-responder가 해결한 장애 패턴을 feedback 메모리로 축적

🛡 Advisor Tool 심화

현재 agents/_shared/advisor-guard.md는 기본 advisor만 주입. 확장 포인트:

  • C-Level별 전문 advisor (CPO advisor — PRD 완전성 체크, CSO advisor — OWASP 커버리지 체크)
  • 실시간 policy advisor — GDPR/라이선스 위반 의심 시 사용자 체크포인트 강제
  • 비용 advisor — finops-analyst가 제안한 최적화가 실제 적용될 때 영향 모델링

📊 Telemetry / 대시보드

.vais/event-log.jsonl에 감사 로그가 쌓이고 있지만 아직 시각화가 없음.

  • C-Level별 평균 처리 시간·리루프율
  • 서브에이전트 병렬도·토큰 사용량
  • Gate 실패 패턴 분석 → 자주 막히는 단계 개선
  • /vais dashboard가 있긴 하지만 정적 HTML 수준 → 실시간화

🧪 테스트 체계

현재 scripts/vais-validate-plugin.js로 구조 검증은 되지만, 에이전트 행동 테스트는 없음.

  • Golden path 시나리오 (S-1 ~ S-10)에 대한 end-to-end 재현 테스트
  • sub-agent description 품질 자동 평가 (3인칭/what/when 존재 여부)
  • 페르소나별 대화 시뮬레이션

🌏 다국어 · 브랜드

  • 현재 모든 내부 문서가 한국어/영어 혼재 — i18n 체계
  • CBO copy-writer를 활용한 일관된 브랜드 톤/voice 가이드
  • 공개 랜딩 페이지·튜토리얼 영상

🔄 외부 스킬 흡수 (S-9)

CEO의 absorb 모드 + skill-creator는 있지만, 마켓플레이스 다른 플러그인을 VAIS 스타일로 변환하는 파이프라인이 아직 자동화 부족. 기회:

  • 외부 agent md → VAIS C-Level 하위로 자동 배치
  • 중복 기능 감지 (이미 있는 sub-agent와 겹치는지)
  • 흡수 후 references/_inbox/에 감사 로그 남기고 자동 PR

4.3 기여하려면 어디부터?

🟢 Easy · 문서
각 sub-agent md의 description 개선 (3인칭·what·when 보강). skill-validator가 지적하는 항목부터.
🟡 Medium · 스크립트
doc-validator.js에 새 규칙 추가, dashboard.html 실시간화, pre-commit hook 확장.
🔴 Hard · 구조
MCP 서버 구현, 프로젝트 레벨 memory 시스템, advisor 심화.
